# cap-shape — 收敛阶段：模糊点子 → 获批规格

把"想做点什么"变成一份**获批的、无歧义、能被 cap-plan 直接吃进去**的规格。

> **铁律(HARD-GATE)**:spec 拿到用户批准之前,**不写任何实现代码、不搭脚手架、不调任何实现类阶段**。
> 需求看起来越简单越危险——"简单"往往是没被检视的假设,返工最多都出在这。规格可以很短(简单需求几句话),
> 但**必须呈现并获批**。

> **本阶段是 cap-flow 派来的收敛器**:知识与状态都是纯文件,引擎 = 一个能 Read/Edit/Bash/Grep 的模型。
> 唯一的终态 = 产出获批 `spec.md` + 把交接写回 STATE,由 cap-flow 路由到 `cap-plan`。

## 0. 可移植前置（每次入口先做）

> **共享 references 的位置**:本文引用的 `role-routing.md`、`divergence-frames.md`、`roles/<role>.md`、
> `web-review/playbook.md` 物理上都在编排器目录 `cap-flow/references/` 下,不在本阶段目录里。解析路径一律
> 指向 `cap-flow/references/...`(相对 skills 根)或经软链接定位,别当相对本目录读。

本阶段要在 Claude 和 Codex 上都能跑。两条降级范式贯穿始终:

### 0.1 交互降级 —— 纯文本编号选项（默认）
凡向用户提问(澄清、选方案、确认范围、过门),**默认用纯文本编号列表**,不硬依赖任何富交互控件:

```
需要你定一个点:
  1) 选项 A —— 说明 / 取舍
  2) 选项 B —— 说明 / 取舍
  3) 其它(你来描述)
回个编号即可(或直接补充文字)。
```

宿主有结构化提问控件可以用,但**回退路径永远是上面这种编号文本**。若用户选"其它"又没填内容 →
输出一行 "你想补充什么?" 然后**停下等**,别重问、别自答。

### 0.2 并行降级 —— 能并行就并行，不能就串行
本阶段的外部研究(§3 可选)若要多源调研:探测宿主有无并行能力——有 → fan-out 子调研,各写各的临时
笔记;无 → 串行 inline 跑同一份 playbook。**任何时候不得两个写手同写 `STATE.md` / `spec.md`**(单写者)。

## 1. 入口条件

进入本阶段应满足(cap-flow 保证;独立调用时自检):

| 条件 | 要确认的事 |
|---|---|
| 有 `<target-repo>/.cap/PROFILE.md`,或确认是 greenfield | 有 PROFILE 即可;brownfield 缺 PROFILE 应先 `cap-map` |
| `STATE.stage` 为 `shape`,或这是一个全新特性的开始 | 读 `.cap/STATE.md`;无 STATE = 新特性,本阶段初始化一份 |
| 手头有一个待澄清的"点子 / 需求 / 变更意图" | 来自用户原话 |

入口动作:先看一眼 `<target-repo>/.cap/` 有没有 PROFILE / STATE / 既有 spec,再:

- 读 `PROFILE.md` → 拿技术栈、约定、surface-map(模块 / glob / 默认角色 / 验证项),当 Explore 的地图。
- 读 `STATE.md`(若有)→ 看 `Decisions log` 里有没有相关旧决策,**别重复问已定的事**。

### 1.0 入口先做：改动面 → 角色 + 验证项 预解析
即便本阶段还没真动代码,也要**先定本特性"打算动哪块 surface"**,因为它决定:

- 要不要在 spec 里**前置 eval 契约**(动 AI / 模型 / 策略 → 必须,见 §5);
- 后续 build / verify / review 会装哪些角色卡、跑哪些验证项。

做法(读编排器的路由规则,不内联整张表):

1. 读 `cap-flow/references/role-routing.md`(路由规则)+ `PROFILE.md` 的 surface-map。
2. 若已有改动:`git -C <target-repo> diff --name-only HEAD` → 按 `resolve(diff × surface-map × routing)`
   算出预期活跃角色 / 验证项。
3. 若全新、尚无改动:让用户用编号文本声明**本特性要动哪个 surface**(前端 / 移动 / API / AI 策略 /
   数据管道),据此预取默认角色 + 验证项。
4. **关键派生**:若解析出的验证项含 `model`(即触及 `models/ strategy/ prompt/ ai/ evals/` 等 AI 面)→
   **本次 spec 必须包含 §5 的 eval 契约**。

把预解析结果记进本阶段 STATE 快照(`verify-checks:` / `Active roles`),**仅作交接提示,不当权威**
(build / verify 会重算)。

5. **定 `work-type` 流程画像**(写进 STATE):默认 `feature`;若本特性是"改造遗留 / 补 AI-readiness" →
   `remediation`(spec 一段话、契约多 N/A、下游走 L1 + 可跳 TDD,但 review / verify 不放松);紧急修 →
   `hotfix`。这是下游各阶段"走多重"的中央旋钮。

## 2. 步骤流程

按顺序执行,每步建议开一个 todo 跟踪:

```
1 Explore(摸三现状)
2 Scope guard(拆超大需求 / 防蔓延)
3 Clarify(一次一问,澄清意图)
4 Approaches(2-3 方案 + 推荐 + 取舍)
5 Design(分节呈现,逐节获批)
6 Eval 契约(仅 AI / 模型 / 策略:定 rubric / 数据集 / 阈值)
7 文档矩阵 → 写 spec.md(+ 必要时 ADR)
8 Spec 自检(占位 / 矛盾 / 范围 / 歧义 四扫,就地修)
9 用户复核门(用户看过写好的 spec)
10 交接 → 写回 STATE,路由 cap-plan
```

### 2.1 Explore — 摸清三现状
别急着提方案,先弄清三件事:

- **代码现状**:涉及哪些模块?现有类型 / 接口 / 数据流?`grep` + `find` 快速定位;范围 > 3 模块时按 §0.2 展开。
- **文档现状**:有无相关 spec / ADR?过期没?(看 `docs/specs/`、`docs/adr/`、`.cap/spec.md`)
- **测试现状**:有无覆盖相关路径的测试?(这决定 plan / build 的 TDD 起点)

把发现浓缩成几句"现状摘要",作为后续提问与方案的依据。

### 2.2 Scope guard — 防爆量 / 防漂移
- **拆超大需求**:需求里塞了多个独立子系统("一个带聊天 + 存储 + 计费 + 分析的平台")→ **先 flag 并分解**,
  别在一个注定要拆的项目上耗澄清问题。每个子项目各走一遍 shape→plan→build。
- **防 scope creep**:澄清阶段只澄清"**怎么实现已在范围内的东西**",不引入新能力。用户提的新能力 →
  记进 spec 的 **Deferred Ideas** 区,不丢、不就地做:

  ```
  「X」是一个新能力——它该是自己的一个特性。
  我先把它记进 Deferred,本次聚焦在【当前特性域】。
  ```

### 2.3 Clarify — 一次一问（核心纪律）
- **一条消息只问一个问题**。一个话题要多问就拆多条,不要一次轰炸。
- **优先给编号多选**,开放式也行。
- 聚焦:**目的 / 约束 / 成功标准**。
- **你是 builder,用户是 visionary**:问"愿景与实现取向",**不要问**用户:代码库模式(你自己读代码)、
  技术风险(你识别)、实现路径(plan 决定)、成功度量(从工作推断)。
- **灰区要具体化**:**禁用泛标签**("UI / UX / 行为")。给本特性生成具体灰区:
  - "用户认证" → 会话处理 / 错误响应 / 多设备策略 / 找回流程
  - "整理图库" → 分组依据 / 重复处理 / 命名约定 / 文件夹结构
- **空答处理**:回答为空 → 重问一次;仍空 → 用编号文本请用户选。绝不带着空答继续。

### 2.4 Approaches — （可选发散 →）2-3 方案

**何时引入发散**:问题已理解(2.1/2.3 完)、**要选方案前**,先过 `cap-flow/references/divergence-frames.md`
的入口三问:

- **开放式**(多个可行答案,非唯一教科书解) **且 高风险**(架构 / 公开 API / 命名 / schema,选错代价高)
  **且 开放措辞**(用户没说 quick / standard / canonical / just)→ **三者全满足才跑发散**;
- 任一不满足(尤其用户措辞收敛、或就是常规 CRUD)→ **跳过发散,直接给常规 2-3 方案**。

```
2.3 Clarify 完(问题已理解)
   ↓
2.4 入口门(divergence-frames 三问)
   ├─ 全 yes → 发散:多框架爆点子(能并行就并行)→ 打分 / 聚类 / 标陷阱 → 深化 top3
   │            → 存活的"非显而易见但可行"候选,喂进下面的收敛
   └─ 否则  → 直接进收敛
   ↓
收敛:提 2-3 方案,带取舍,领头给推荐 + 理由(对话式,用户决策)
```

- 发散只**喂候选**;最终仍收敛到 **2-3 方案**带取舍、**领头给你的推荐 + 理由**、对话式让用户决策(不替他定)。
- 发散**可选 + 门控 + 可移植**(无并行则串行跑几个框架);贵,默认不跑,只在真正开放 / 高风险设计决策时点用。

### 2.4b 范围塑造（可选，门控同 §2.4）
收敛到方案前,对本特性显式做一次范围决策:

1. **10 星框架**:先问"这功能的 10/10 版本长什么样?",再决定**这次做到几星**——别默认抄第一个想到的范围。
2. **挑战前提**:这事到底该不该做?有没有更小的切法 / 更好的框架达成同目标?
3. **显式选姿态**:给本特性标一个——**EXPAND**(值得做大)/ **HOLD**(刚好)/ **REDUCE**(剥到本质 MVP)/
   **SELECTIVE**(部分加、部分减),一句话说为什么。

> 与 scope-guard 互补:guard 拦"不该加的",本步决定"该往哪个方向塑形"。结论写进 spec §4「方案与决策」。

### 2.5 Design — 分节呈现，逐节获批
- 自认为理解了再呈现设计。**每节按复杂度伸缩**:直白的几句话,微妙的可到 200-300 字。
- 覆盖:架构 / 组件 / 数据流 / 错误处理 / 测试策略。
- **每节问一句"到这里对不对?"**,获批再继续(增量确认)。某节不对就回头澄清。
- **为隔离与清晰而设计**:拆成单一职责、接口清晰、可独立理解与测试的小单元;每个单元能答"它做什么 /
  怎么用 / 依赖什么"。
- **在既有代码库里工作**:先探索现有结构、沿用既有模式;只对"影响本次工作的既有问题"做定向改进,**不提无关重构**。
- **对抗性自我证伪**:涉及**跨模块架构 / 数据模型变更 / API 契约**这类高代价决策时,**定稿前先自我证伪一轮**——
  主动问"这个设计哪里会塌?有没有反例 / 边界 / 更简方案?",而不是事后 review 才发现方向错。比事后返工便宜得多。
  **触发**:命中 architect 角色或动核心数据结构 / 契约时;轻量特性跳过。结论(及被否方案)写进 spec §4。

### 2.6 Eval 契约（★ 仅当本特性触及 AI / 模型 / 策略）
见 §5。AI 工作的质量标准**必须在 spec 阶段定死**,由后续 `cap-verify` 的 `model` 验证项执行。非 AI 特性跳过。

### 2.6b 设计契约（★ 仅当本特性触及 UI / 前端面）
**与 §2.6 完全对称**——eval 契约管"怎么算 AI 答得好",设计契约管"怎么算 UI 做得好"。
**触发**(机器可判):§1.0 解析出的活跃角色**含 `design`**(diff 命中前端 / 移动面)→ **本特性必须产出或更新
`<target-repo>/DESIGN.md`**。非 UI 特性跳过。

**为什么必须前置**:`design` 角色卡反复声明"判定以 `DESIGN.md` 为准,没有就退回通用原则"——可没人产出它。
本步就是**那个产出者**,让 design 卡有项目专属"设计宪法"可校准。

**DESIGN.md 内容**:
- **风格方向**:选一个明确方向(editorial / brutalism / 玻璃拟态 / Swiss…),**禁"clean minimal"这种空话**。
- **色板 token**(语义化)/ **字体配对**(≤2-3 家族 + 配对理由)/ **间距标尺**(4 或 8px 网格)。
- **设计系统**:组件约定、圆角 / 阴影规范、**交互态**(hover / focus / active + focus-visible)、动效(只动
  compositor 属性)。
- **响应式断点** / **a11y 基线**(对比 ≥WCAG AA、键盘可达、reduced-motion)。

**两条撰写纪律**:
- **具体参照物 > 形容词堆砌**:用一个具体参照物锚定风格(如"1970 年代某老牌大学的研究生讲义"),比
  "modern / clean / premium"携带的信息多一个数量级——形容词描述一个区域,参照物描述一个点。**负约束随参照物
  自带**(讲义不会发光 / 渐变,不必逐条列);Do's / Don'ts 用来收紧参照物覆盖不到的边角,不是替代它。
- **散文为主、token 为辅**:token(色值 / 字号 / 间距)是给实现者校准的 *context*,不是渲染指令;DESIGN.md 的
  判定力主要来自散文把"为什么是这个值、何时怎么用"讲清楚,而非堆砌精确数值。

**轻重自适应**:首个 UI 特性 → 建完整 DESIGN.md;后续 UI 特性 → 只**增补 / 收紧**新模式,不重写。UI 微调
(已在 DESIGN.md 范围内)→ 无需改。

**闭环**:spec 产 DESIGN.md → build 照它实现 → design 卡校准到它(不再退回通用原则)→ review 的 design 透镜 +
journey:Web 据它验。

### 2.7 文档矩阵 → 写 spec.md
用文档选择矩阵决定写什么、写哪:

| 变更类型 | 写什么 | 放哪里 |
|---|---|---|
| 核心数据结构变更、模块拆分合并、API 契约变更 | **ADR** | `docs/adr/`(同时在 spec 里引用) |
| 用户可感知的功能变更(页面 / 流程 / 交互) | **Spec** | `.cap/spec.md`(主产物) |
| 局部实现细节、UI 微调 | 更新已有文档 | 找到对应 spec 追加 / 修订 |
| 纯 bug fix(行为不变) | 不改文档 | —(且更可能该走 cap-build 的调试子循环) |

主产物始终是 `<target-repo>/.cap/spec.md`(统一格式见 §4)。涉及架构变更时**额外**写一份 ADR 并在 spec 里链接。

**ADR 模板**(`docs/adr/NNNN-<title>.md`):
```markdown
# ADR-NNNN: <决策标题>
Status: proposed | accepted | superseded-by ADR-XXXX     Date: <caller 传入>
## 背景 / 问题      —— 为什么要决策,约束是什么
## 决策            —— 选了什么(一句话能说清)
## 备选与否决理由   —— 还考虑过什么、为何不选(对抗性证伪 §2.5 的产物落这)
## 退化 / 保留      —— 这个决策让什么退化、又保住了什么
## 架构影响 / 后果  —— 正负后果、blast-radius、可逆性(architect 视角)
```
> ADR 是**架构决策的不可变记录**(append-only,改决策 = 写新 ADR 并 supersede 旧的),供 architect / review
> 回溯"当初为什么这么定"。

### 2.8 Spec 自检（写完后用新视角扫一遍，就地修）
1. **占位扫描**:有 "TBD / TODO"、空节、模糊需求?→ 补全。
2. **内部一致**:各节是否互相矛盾?架构是否与功能描述对得上?
3. **范围检查**:是否聚焦到一个实现计划可承载?还是需要分解?
4. **歧义检查**:任何需求能被两种解读?→ 挑一种、写明确。

就地修完即可,不必再 review 一轮。

### 2.9 用户复核门
自检通过后,请用户复核**写好的 spec 文件**:

```
Spec 已写入并(如适用)提交到 `<path>`。请你过一眼,
有要改的告诉我;没问题我就进入 cap-plan 出实现计划。
(spec 长 / 多节多表,逐条聊批太累?说「网页审」,我渲染成可划词批注的本地网页让你标。)
```

> **主动提醒(硬性告知 · 一次性)**:上面那行括号「网页审」选项**必须出现在本复核提示里**——这是每次
> spec 门的**告知义务**,不是可选修辞,**不得**以"默认聊天更快 / spec 简单"为由省略。选不选仍归用户:
> 用户不点「网页审」即默认走编号文本聊天批(更快)。

等待用户响应。要改 → 改完**重跑 §2.8 自检**。**只有用户批准后才前进**。

**可选 · 网页划词批注复核**:spec 长 / 结构复杂(多节多表 / 跨域契约)、聊天逐条批太累时,可把 spec 渲染成
「划词加批注」的本地网页让用户标注,再统一改回——见 `cap-flow/references/web-review/playbook.md`。它只升级
"意见怎么喂回来"的机制,不改"批准前不前进"的纪律。需要**实时双向**(提交即自动回流、改完页面自刷新、跨引擎)→
同 playbook 的 Live mode。

## 3. 分级 Explore（外部研究可选）

一条**分级 Explore**(轻→重,按需升级):

| 级别 | 做什么 | 何时用 |
|---|---|---|
| L1 本地三现状 | grep / find 摸代码 / 文档 / 测试现状(§2.1) | **总是** |
| L2 轻量 scout | 在现状基础上识别灰区、可复用资产、决策点 | 需求有多种实现走向时 |
| L3 有界外部研究(可选) | 调 unknown-unknowns:标准架构模式 / 标准技术栈 / 常见坑 / **Don't-Hand-Roll**(哪些不该手搓)/ SOTA vs 训练直觉 | 引入不熟的技术 / 第三方依赖 / 新框架时 |

**L3 关键提问**:核心问题不是"该用哪个库",而是"**我不知道自己不知道什么**"。研究产出要**规定性**("用 X"),
不是探索性("可考虑 X 或 Y"),并附 **confidence 分级**(高 / 中 / 低,关键论断多源核实)。

**L3 可移植化**:有并行能力就 fan-out 子调研,没有就串行;研究结论**直接并入 spec 的"技术约束 /
Don't-Hand-Roll"节**,不另起运行时产物。

## 4. 读写哪些 .cap/ 文件

| 文件 | 读 / 写 | 内容 |
|---|---|---|
| `.cap/PROFILE.md` | **读** | 技术栈 / 约定 / surface-map(Explore 地图、路由输入)。本阶段不写 PROFILE。 |
| `.cap/STATE.md` | **读 + 写**(经 cap-flow,单写者) | 读 Decisions log 防重问;写回 stage / gates / decisions / next(§6) |
| `.cap/spec.md` | **写(主产物)** | 统一格式见下 |
| `<target-repo>/DESIGN.md` | **写(条件)** | 仅 UI / 前端面工作时(活跃角色含 design);产出 / 更新设计契约,§2.6b |
| `docs/adr/*.md` | **写(条件)** | 仅架构变更时;spec 内链接 |

### 4.1 统一 spec.md 格式

```markdown
# Spec: <feature/topic>

> Date: <由 caller 传入的日期>
> Status: draft | approved
> Target surface(s): <web-frontend / api / ai-strategy / ...>  # 来自 §1.0 路由解析
> Active roles (anticipated): <server-dev, qa, ...>
> Verify checks (anticipated): <logic, journey:Web, model, ...>

## 1. 问题 / 目标
为什么做?要达成什么用户 / 业务结果?

## 2. 非目标(YAGNI)
明确不做什么,防 scope creep。

## 3. 现状摘要(Explore 产出)
代码 / 文档 / 测试三现状的浓缩。

## 4. 方案与决策
选了哪个方案、为什么(含被否方案的取舍)。架构变更链接到 ADR。

## 5. 设计
架构 / 组件 / 数据流 / 错误处理 / 测试策略(逐节获批后的最终版)。

## 6. 怎么算 done(前置验收)
在改任何代码前就对齐的验收标准 + 验证命令(喂给 cap-plan / cap-verify)。

## 7. Eval 契约(仅 AI / 模型 / 策略工作;否则写 "N/A")
见 §5 —— rubric / 数据集 / 阈值 / 测量法 / guardrail-vs-flywheel。

## 7b. 设计契约(仅 UI / 前端工作;否则写 "N/A")
见 §2.6b。产出 / 更新 `<target-repo>/DESIGN.md`(风格方向 / 色板·字体·间距 token / 设计系统 + 交互态 /
响应式 + a11y);本节只需链接到 DESIGN.md 并标注本次新增 / 收紧了哪些设计决策。

## 8. Deferred Ideas(结构化延后)
被挡下的 scope creep / 大点子。每条带 Why + Trigger(何时再捡起)+ Breadcrumbs(相关文件 / 决策)。

## 9. Canonical refs(强制累积)
本特性依赖 / 参考的源文件、文档、外部资料路径(每轮讨论持续累积,不丢线索)。
```

> **结构化 deferral**:Deferred 不是一句话扔角落没人看,而是带 **Why(为什么重要)+ Trigger(什么条件下
> 再做)+ Breadcrumbs(去哪找细节)**,避免 context rot。

## 5. ★ Eval 契约前置（AI 工作在 spec 阶段定质量门）

当 §1.0 路由解析出 `model` 验证项(本特性触及 `models/ strategy/ prompt/ ai/ evals/ agents/` 等 AI 面),
**spec 必须把 AI 的质量标准在动代码前定死**,后续 `cap-verify` 的 `model` 验证项据此执行。

### 5.1 为什么必须前置
AI 系统**非确定性**:同样输入不保证同样输出。单测 / 集成测不足以判 AI 工作"够不够好"。质量标准若不在
spec 阶段定,到 verify 阶段就会临时拍脑袋。

### 5.2 spec 里要写清的 5 件事（→ §4.1 的 "Eval 契约" 节）

1. **系统类型 + 关键失败模式**:这是什么 AI 系统(RAG / 对话 / 结构化抽取 / 多步 agent / 分类)?列
   **3-5 个绝不能出错的行为**(决定主导 eval 维度)。
2. **测量法**(三选优先级,可组合):
   - **代码指标优先**(确定性:JSON 合法、必带免责声明、性能阈值、分类 flag)——快 / 便宜 / 可靠,先用这个。
   - **LLM judge**(主观质量:语气 / 推理 / 升级判断)——**必须先用人工校准过**才可信。
   - **人工评估**(金标准,不可规模化)——用于校准、边缘、抽样、高风险。
3. **rubric**(每个被测维度必须有):
   - 测的维度名;
   - 1 / 3 / 5 分(或 pass / fail)各代表什么——**领域特化,不要泛指标**("helpfulness"在房产 = 清晰总结
     房源,在医疗 = 知道何时**不**回答);
   - 可接受 vs 不可接受 的领域示例。
   没有 rubric,LLM judge 只产噪声不产信号。
4. **reference dataset**:起步 **10-20 个高质量样本**(不是 200 个平庸的);覆盖关键成功场景 / 常见工作流 /
   已知边缘 / 历史失败模式;最好由领域专家标注。写明数据集**存放位置**(供 verify 读取)。
5. **阈值 + verdict 规则**:各维度加权 + 通过阈值(如"加权分 ≥ 4.0/5 且无任一维度 < 3 才算过");并区分
   **guardrail(出错即灾难→在线实时拦截)vs flywheel(不灾难→离线批量驱动迭代)**。

### 5.3 spec → verify 契约接口
在 spec 的 Eval 契约节里**显式写明三个位置**,让 `cap-verify` 的 `model` 验证项无歧义读取:
- rubric 定义在哪(spec 本节 / 独立文件路径);
- reference dataset 在哪(路径);
- 阈值 / verdict 规则是什么。

> **护城河钩子(F1 proof-of-value)**:reference dataset 与 verdict 规则同时也是"接入中心知识库后模型有没有
> 变准"的量化基线——`cap-verify` 的 `model` 验证项对接 server 端 eval 打分(如文件预测 F1),把这套 spec
> 契约变成可回归的证据。所以数据集与阈值要写成**可复跑**的,不是一次性拍脑袋。

> 这就是"标准在 spec 定、执行在 verify 做"的契约——verify 不需要回放讨论历史即可执行评估。

## 6. 出口门控 + 写回 STATE

### 6.1 出口门控（双门，全过才离开本阶段）
- [ ] **HARD-GATE 守住**:批准前未写任何实现代码。
- [ ] **设计逐节获批**:每节都拿到用户确认。
- [ ] **spec.md 已写**,格式符合 §4.1,无占位(§2.8 自检通过)。
- [ ] **若 AI 工作**:§5 Eval 契约已填(rubric / 数据集 / 阈值 / 契约位置齐全),否则该节写明 "N/A"。
- [ ] **若 UI / 前端工作**(活跃角色含 design):§2.6b 设计契约已产出 / 更新 `DESIGN.md`(风格方向 / token /
  交互态 / a11y 齐全),spec §7b 链接到它,否则该节写明 "N/A"。
- [ ] **用户复核门通过**:用户看过写好的 spec 并批准(§2.9)。

任一未过 → `status: gated`,停在闸口,用编号文本列出待办项,**不前进**。

### 6.2 交接给 cap-flow（HANDOFF → STATE）
spec 获批后,输出 `## HANDOFF` block;cap-flow 读取后作为单写者写回 `<target-repo>/.cap/STATE.md`
(schema 见 cap-flow / STATE 模板;时间戳由 caller 传入)。独立直调本阶段且没有编排器时,也必须先产同一
`## HANDOFF`,再作为单写者应用到 STATE:

```markdown
## HANDOFF
stage: shape
status: in-progress            # 或 gated(未过复核时)
verify-checks: [logic, journey:Web, model]   # §1.0 预解析的快照(非权威,build 会重算)
active-roles: [server-dev, qa]
changed-files:
- (none yet — target surface: <surface>)
gates-passed:
- shape: spec.md 已获批(含 AI 工作的 eval 契约,若适用)
decisions:
- <date> 选 <方案> 而非 <备选>,因为 <理由>
- <date>(若 AI 工作)eval rubric / 数据集 / 阈值已在 spec §7 定义
next-action: -> cap-plan
```

写完向用户报告 stage / status / next,并据 status:
- `in-progress` → 提示可进入 `cap-plan`(或本会话直接续)。
- `gated` → 停在复核闸口,列出待批项。

## 8. 不要做的事（反模式）

- ❌ 批准前写实现代码 / 脚手架 / 调实现类阶段(违反 HARD-GATE)。
- ❌ 一条消息塞多个问题(违反一次一问)。
- ❌ 用泛标签(UI / UX / 行为)当灰区,不具体化。
- ❌ 替用户做愿景决策;或问用户技术风险 / 实现路径 / 成功度量(那些你自己定 / 识别)。
- ❌ 引入范围外的新能力(应记 Deferred,结构化延后)。
- ❌ AI 工作不定 eval 契约就放行到 plan(违反 §5 前置)。
- ❌ 把 Deferred 写成无人看的一句话(必须带 Why + Trigger + Breadcrumbs)。
- ❌ 并发写 STATE.md / spec.md(违反单写者)。
- ❌ 在本阶段拆任务 / 写测试 / 跑验证(那是 plan / build / verify 的活)。
