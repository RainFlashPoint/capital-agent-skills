# Complexity Routing — L1–L4

复杂度等级决定研发流程的最短合法路径，不改变所有等级都必须遵守的结果硬门。

## 1. 判定顺序

在入口根据以下信号取最高命中等级：

1. 命中资金、安全、用户数据、生产发布、原始 SQL 或不可逆迁移 → 至少 L4。
2. 涉及外部接口契约、支付渠道、跨模块、数据库、并发或多个服务 → 至少 L3。
3. 需要维护、测试或少量跨文件改动，但没有高风险信号 → L2。
4. 单文件、低风险、契约已知且验证命令明确 → L1。

用户可以手动升级等级；不能绕过高风险信号强制降级。若无法判断，取较高等级并记录原因。

## 2. 最短合法路径

### L1 — 小问题

适用：单文件低风险修复、文案/配置、明确测试补充，无外部契约或高风险路径。

```text
intake → build → verify
```

不要求独立 `spec.md` 或 `plan.md`，但必须记录任务边界、改动文件和验证证据。

### L2 — 普通问题

适用：现有模式内的小功能、少量文件改动、无架构和外部契约变化。

```text
shape-lite → build → verify → review-lite
```

`shape-lite` 产出短任务契约；只有任务确实需要拆分时才生成独立 `plan.md`。

### L3 — 复杂问题

适用：跨模块、外部接口、支付渠道、数据库、多个执行依赖或项目上下文不完整。

```text
map → shape → plan → build → verify → review
```

已有且未漂移的 `PROFILE.md` 可以复用，不能为了形式重复 map。

### L4 — 高风险问题

适用：资金、安全、生产、数据迁移、核心架构或大范围重构。

```text
map → shape → plan → build → verify → review → release
```

必须保留独立设计、验证、评审和发布门，不允许自治流程跳过人工决策。

## 3. 全等级硬门

无论 L1–L4，以下条件都不能由 Agent 自行豁免：

- 禁止路径不得修改；
- 没有证据不得猜测外部契约；
- 必要编译或测试失败不得标记完成（验证失败不得标记完成）；
- 没有真实 Commit 和改动清单不得交付；
- 未验证项必须显式记录；
- 安全、资金和生产风险必须升级人工。

## 4. 产物最小化

- L1：任务契约 + 验证证据即可；
- L2：短 spec/plan + 验证证据；
- L3：`PROFILE/spec/plan/verify/review` 按实际命中生成；
- L4：完整主线产物和发布证据。

复杂度路由的目标是减少无效文档，不是降低质量门。
